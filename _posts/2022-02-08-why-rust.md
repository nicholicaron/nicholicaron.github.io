A short piece on how Rust helped me fall in love with computers.

I came to programming through mathematics — discrete structures, formal logic, proof techniques. When I first encountered Rust, something clicked in a way that no other language had managed. It wasn't just that Rust was fast, or that it prevented memory bugs. It was that Rust's type system felt like a proof assistant for systems programming. The compiler doesn't just check your code — it reasons about it.

## What Drew Me In

- **Ownership and Borrowing** — Memory safety without garbage collection. The borrow checker is strict, but the discipline it enforces produces code that's correct by construction rather than by testing.
- **Zero-Cost Abstractions** — Iterators, closures, and generics that compile down to the same machine code you'd write by hand in C. You don't pay for what you don't use.
- **Algebraic Data Types** — `enum` and `match` make invalid states unrepresentable. Pattern matching exhaustiveness checking means the compiler catches cases you forgot.
- **The Ecosystem** — Cargo, crates.io, and a community that genuinely cares about documentation and correctness.
- **Fearless Concurrency** — The type system prevents data races at compile time. Not at runtime. Not with locks you might forget. At compile time.

## The Deeper Point

Rust didn't just teach me a new syntax. It changed how I think about software — about what it means for a program to be correct, about the relationship between types and proofs, about why constraints are liberating rather than limiting. The borrow checker is frustrating at first, but once you internalize its model, you start seeing memory bugs in every C program you read.
